Lab 6

There are two main ways of debugging, print statements and utilizing the Eclipse debugger. We're going to try out both today and learn how to use them. Let's start by creating a new Eclipse class to work on this lab. You can follow what we did on Lab 1 or Lab 2 to do this. Make sure to call it lab6. You should end up with this:

Copy the folowing code inside the public class lab6 { in the project we just created:

public static int mystery(int n) {
    int y;
    int x = 1;
    int total = 0;

    while ( x <= 10 ) {
       y = x * x;
       total += y;
       ++x;
       } // end while

    return total;

System.out.println(mystery(15));

What does this function do? Lets approach it using print statements first.

Part 1: Print statements

The most common way, and what you guys have probably been using, is to use print statments such as System.out.println(x); to debug our code (where is x is a variable of interest). We just place a verity of print statements around the code to in an attempt to figure out what the code does. Take 2 minutes and places some print statements around the code in order to try and understand it. Do not spend more than 2 minutes on this, the second part of this lab should make it a lot easier to debug.

Part 2: Eclipse Debugger

In the second part of the lab we're going to use a much more powerfull tool built into eclipse. To start copy the folowing code inside the public class lab6 { in the project we just created:

Start the Debugger: Clicking the ant icon (see below) starts the debugger:

You can see that clicking it does nothing right now :(, the code will run normally as if you clicked the run icon. This is because we have set no breakpoints. Breakpoints (see blue dots below) can be set by clicking the space between the lines of code and the edge of the screen:

What do breakpoints do? They tell the debugger when to stop! Now if we click the ant icon, the debugger will stop, and it should look like this:

That's because we stopped at the breakpoint we set. [Try moving around the breakpoint and see how this changes, just press the red stop button to return] Now once we're in the debugger we have 3 different ways of moving around, as seen here:

  1. The first one: Step Into: Steps into the function, for example on line 8, if we pressed step into, we would dive into the function Foo starting from line 13.
  2. The second one: Step Over: Steps over the function, for example on line 8, if we pressed step over, we would not dive into the function, and continue on line 8.
  3. The third one: Step Out Of: Steps out of the function, for example, if we were in the function Foo, and pressed step out of, we would return back into the main function.

Now imagine if we are done with debugging, and want to return back to the main function, without executing the remaining lines in Foo. We could click Step Out Of to do this:

Notice how x is no longer in scope, and can't be seen in the variables. If we step over to the next line, result would come into scope and we would be able to see what the function foo returned:

This is it, these are the basis of the debugger, it helps us move around code in order to see what values variables have and we can use this information to change and modify our code! *wow*
